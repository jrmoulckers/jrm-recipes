# Internationalization (i18n) workflow

Heirloom ships message catalogs with [next-intl](https://next-intl.dev). The UI
renders whichever catalog the request resolves to (cookie → `Accept-Language` →
default), with **English as the source of truth**. This doc covers how to add or
change copy and how to keep the translated catalogs honest.

## Where things live

| Thing                       | Path                                                         |
| --------------------------- | ------------------------------------------------------------ |
| Locale catalogs             | `src/messages/<locale>.json`                                 |
| Supported locales / default | `src/config/i18n.ts` (`SUPPORTED_LOCALES`, `DEFAULT_LOCALE`) |
| Runtime wiring              | `src/i18n/request.ts`                                        |
| Parity validator            | `scripts/i18n-validate.mjs` (`pnpm i18n:validate`)           |
| Pseudolocale generator      | `scripts/i18n-pseudo.mjs` (`pnpm i18n:pseudo`)               |

Catalogs are nested JSON grouped by feature namespace (`nav`, `cook`,
`recipeEditor`, …). Values are [ICU message strings](https://next-intl.dev/docs/usage/messages),
so placeholders like `{count}` and `{count, plural, …}` are part of the contract.

## Adding or changing copy

1. Add the key(s) to **`src/messages/en.json`** first — English is the source.
2. Mirror the same keys into every other catalog (`es`, `de`, `ar`). Until a real
   translation exists, copy the English value so the key is present and the app
   never renders a raw key.
3. Reference the key in components via `useTranslations`/`getTranslations`.
4. Run `pnpm i18n:validate` and `pnpm typecheck` before pushing.

Adding a **new locale**: create `src/messages/<locale>.json`, add the id to
`SUPPORTED_LOCALES` (and the endonym/OpenGraph maps) in `src/config/i18n.ts`, and
import it in `src/i18n/request.ts`.

## Validating catalog parity

```bash
pnpm i18n:validate        # human-readable report, exits non-zero on drift
node scripts/i18n-validate.mjs --json   # machine-readable
```

The validator reads `SUPPORTED_LOCALES`/`DEFAULT_LOCALE` from
`src/config/i18n.ts`, flattens each catalog to dot-delimited keys, and compares
every locale against the source. It flags, per locale:

- **missing keys** — present in the source, absent in the target,
- **extra keys** — present in the target, gone from the source (stale copy),
- **placeholder mismatches** — a shared key whose ICU arguments differ (e.g. the
  source uses `{count}` but the translation dropped or renamed it).

It exits `1` when any catalog is out of sync, so it works as a local gate and can
be dropped into CI unchanged when Actions is re-enabled (see below).

## Pseudolocalization for QA

```bash
pnpm i18n:pseudo                 # writes src/messages/en-XA.json
node scripts/i18n-pseudo.mjs --check   # verify the file is up to date
```

The generator produces `en-XA.json` from the English source: every letter is
accented, strings are bracketed (`[…•••]`) and padded ~40% longer, while ICU
placeholders and `<tag>` markup are preserved so the message still parses. Use it
to spot copy that never went through i18n (renders without brackets) and layout
that can't absorb longer languages (truncation/overflow).

`en-XA` is **not** in `SUPPORTED_LOCALES` — it ships nowhere and the validator
ignores it. It is also git-ignored (regenerate on demand). To preview it, point
`src/i18n/request.ts` at the generated catalog temporarily during a QA pass.

## CI note

These scripts are intended to run in CI (`pnpm i18n:validate`, and optionally
`node scripts/i18n-pseudo.mjs --check`). GitHub Actions is currently disabled
account-wide for billing reasons, so **no blocking CI gate is wired up yet** —
run the checks locally. When Actions returns, add the commands to the existing
lint/typecheck job; no code changes to the scripts are required.
